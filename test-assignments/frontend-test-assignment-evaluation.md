# Критерии проверки: Frontend Test Assignment (React + Next.js)

## Принцип оценки

Оценивается качество инженерного мышления в продуктово-критичном UI-флоу:  
устойчивость, предсказуемость, безопасность, ясность архитектуры и тестируемость.

Итоговая оценка: **100 баллов**.

## 1) Корректность продуктовой логики — 25 баллов

Проверяем:
- корректная работа формы на happy-path;
- понятная обработка ошибок API (`409`, network error и т.д.);
- корректные UI-состояния (`idle/loading/success/error`);
- предсказуемый UX при повторных действиях пользователя.

Сильный сигнал:
- форма и состояния ведут себя стабильно в типичных и граничных сценариях.

Слабый сигнал:
- пользователь может отправить некорректные данные;
- UI “застревает” в loading/error без восстановления.

## 2) Архитектура и структура кода — 20 баллов

Проверяем:
- разделение ответственности (UI / бизнес-логика / API-слой / state);
- читаемость, именование, композиция компонентов;
- поддерживаемость и возможность расширения.

Сильный сигнал:
- код легко менять без каскадных побочных эффектов;
- state и API не размазаны по компонентам хаотично.

Слабый сигнал:
- “бог-компонент” с смешением всей логики;
- отсутствие очевидных границ между слоями.

## 3) Устойчивость и идемпотентность на клиенте — 15 баллов

Проверяем:
- защита от двойного submit;
- корректная генерация/использование `idempotency_key`;
- восстановление последнего состояния после reload (в заданном окне).

Сильный сигнал:
- повторы запросов не приводят к дублям в UI/логике;
- есть ясная стратегия восстановления контекста пользователя.

Слабый сигнал:
- двойной клик создает дубли;
- перезагрузка ломает пользовательский сценарий.

## 4) Безопасность клиентской части — 15 баллов

Проверяем:
- отсутствие небезопасных паттернов хранения чувствительных данных;
- аккуратная обработка серверных ошибок;
- отсутствие XSS-уязвимых конструкций в рендере.

Сильный сигнал:
- кандидат демонстрирует понимание реального прод-подхода (httpOnly cookies, CSP, и т.д.) даже при mock-auth.

Слабый сигнал:
- токены/секреты кладутся в небезопасное хранилище без объяснения;
- необработанные/внутренние ошибки сервера показываются пользователю.

## 5) Производительность и UX-инженерия — 10 баллов

Проверяем:
- отсутствие лишних запросов и перерендеров;
- корректные loading/disabled/skeleton состояния;
- разумные решения для отзывчивого интерфейса.

Сильный сигнал:
- заметно, что кандидат думает о поведении интерфейса под реальной нагрузкой/сетевыми проблемами.

Слабый сигнал:
- рендеры и запросы не контролируются;
- UX деградирует на медленной сети.

## 6) Тестовая стратегия — 10 баллов

Проверяем:
- тесты покрывают критичные флоу, а не только happy-path;
- есть тест на double-submit;
- тесты помогают ловить регрессии в бизнес-логике UI.

Сильный сигнал:
- тесты короткие, целевые и проверяют инварианты поведения.

Слабый сигнал:
- тесты формальные или отсутствуют на ключевые риски.

## 7) Документация и аргументация решений — 5 баллов

Проверяем:
- понятный `README`;
- явные trade-off’ы (что сделано, что осознанно упрощено);
- план улучшений до production-уровня.

Сильный сигнал:
- кандидат объясняет причины решений, а не только “что сделал”.

Слабый сигнал:
- решения не обоснованы, ограничения не проговорены.

## Красные флаги (существенное снижение оценки)

- Двойной submit приводит к дублям без защиты.
- Критичные ошибки не обработаны, пользователь получает “сырые” server errors.
- Небезопасные паттерны работы с токенами/данными без осознанного объяснения.
- Отсутствуют тесты на ключевой пользовательский флоу.

## Интерпретация итогового балла

- **85-100**: Strong hire (системный frontend-инженер, готов к критичным флоу).
- **70-84**: Hire (хороший уровень, отдельные зоны роста).
- **55-69**: Borderline (нужна дополнительная техническая проверка).
- **<55**: No hire (высокие продуктовые и технические риски).

## Рекомендации интервьюеру

- Попросить кандидата на разборе добавить новое правило (например, комиссии или multi-currency) и оценить, как он адаптирует архитектуру.
- Обсудить, какие улучшения для production он бы сделал первыми и почему.
- Проверить глубину понимания безопасности и производительности на примерах из его решения.
