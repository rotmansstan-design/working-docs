# Тестовое задание: QA Automation Engineer

## Формат и цель

Это короткое тестовое на **2-3 часа**.  
Проверяем QA-мышление: приоритизация рисков, качество тест-дизайна и практичность автоматизации.

## Контекст

Есть флоу вывода средств:
- `POST /v1/withdrawals`
- `GET /v1/withdrawals/{id}`
- `POST /v1/withdrawals/{id}/confirm` (опционально для покрытия)

## Core (обязательно, 2-3 часа)

### 1) API-автотесты (обязательно)

Минимум 4 сценария:
- happy-path create withdrawal;
- `amount <= 0`;
- идемпотентность: тот же ключ + тот же payload;
- идемпотентность: тот же ключ + другой payload.

### 2) E2E (обязательно)

Минимум 2 сценария:
- успешный пользовательский флоу submit;
- ошибка при недостаточном балансе или аналогичный критичный негативный кейс.

### 3) Мини-документация

Сдать `TEST_STRATEGY.md` (до 1 страницы):
- какие риски выбраны как критичные;
- что покрыто сейчас и почему;
- что оставлено вне scope.

### 4) Что сдать

- код автотестов;
- `README.md` (как запускать);
- `TEST_STRATEGY.md`.

## Optional (по желанию, +1-2 часа)

- тест на double-submit в E2E;
- короткий smoke-набор k6/аналог на частые create-запросы;
- пример CI-конфига (GitHub Actions).

## Технические ограничения

- Playwright/Cypress для E2E;
- любой удобный инструмент для API;
- TypeScript предпочтительно.

## Что можно упростить

- один пользователь;
- одна валюта;
- без реальных PSP/крипто-интеграций.
