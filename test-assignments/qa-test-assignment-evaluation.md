# Критерии проверки: QA Automation Test Assignment

## Принцип оценки

Оценивается зрелость QA-мышления:  
насколько кандидат умеет защищать критичные бизнес-флоу через рациональную тест-стратегию, а не просто наращивать число тестов.

Итоговая оценка: **100 баллов**.

## 1) Тест-стратегия и приоритизация рисков — 25 баллов

Проверяем:
- ясный scope и обоснование, что покрывается в первую очередь;
- фокус на финансово-критичных сценариях (withdrawal, идемпотентность, ошибки баланса);
- осознанные компромиссы при ограниченном времени.

Сильный сигнал:
- кандидат связывает тест-приоритеты с бизнес-рисками и стоимостью дефекта.

Слабый сигнал:
- набор тестов выглядит случайным, без приоритизации и rationale.

## 2) Качество E2E-покрытия — 20 баллов

Проверяем:
- покрыт критичный пользовательский путь end-to-end;
- проверены негативные сценарии;
- реализована проверка на double-submit/повторы.

Сильный сигнал:
- E2E-тесты стабильны, не перегружены лишними шагами, дают ценный сигнал.

Слабый сигнал:
- только happy-path;
- E2E хрупкие и флейковые без попытки стабилизации.

## 3) Качество API-автотестов — 20 баллов

Проверяем:
- есть позитивные и граничные сценарии;
- проверена идемпотентность и корректные коды/контракты;
- тесты проверяют бизнес-инварианты, не только статус-код.

Сильный сигнал:
- тесты ловят реальные риски консистентности и поведения API.

Слабый сигнал:
- поверхностная проверка “запрос прошел/не прошел” без инвариантов.

## 4) Устойчивость/нагрузка и инженерный подход — 15 баллов

Проверяем:
- есть разумный smoke-набор для конкурентных/частых запросов;
- понятная цель нагрузочного теста (какую проблему ищем);
- корректная интерпретация результатов.

Сильный сигнал:
- даже небольшой тест показывает системное понимание рисков конкуренции.

Слабый сигнал:
- “нагрузочный тест ради галочки” без метрик и выводов.

## 5) CI/CD и операционная применимость — 10 баллов

Проверяем:
- тесты интегрированы в CI (или есть реалистичный конфиг);
- понятен порядок запуска;
- учтены базовые меры против нестабильности (timeouts/retries/изоляция данных).

Сильный сигнал:
- решение можно быстро встроить в рабочий pipeline.

Слабый сигнал:
- запуск только вручную, без воспроизводимости.

## 6) Качество артефактов и коммуникации — 10 баллов

Проверяем:
- понятный `README` и `TEST_STRATEGY`;
- явно описаны ограничения покрытия;
- есть объяснение, как анализировать падения и искать root cause.

Сильный сигнал:
- документация помогает команде поддерживать и развивать тесты.

Слабый сигнал:
- отсутствует или формальная документация без практической ценности.

## Красные флаги (существенное снижение оценки)

- Нет тестов на критичный withdrawal-флоу.
- Не проверена идемпотентность и поведение при повторах.
- Отсутствуют негативные кейсы для финансово-критичных endpoint.
- Нельзя воспроизвести запуск тестов по инструкции.

## Интерпретация итогового балла

- **85-100**: Strong hire (зрелый QA Automation инженер, мыслит системно).
- **70-84**: Hire (уверенный практик, есть точки роста).
- **55-69**: Borderline (нужна доп. проверка по рисковому мышлению/automation depth).
- **<55**: No hire (высокий риск пропуска критичных дефектов).

## Рекомендации интервьюеру

- На разборе попросить кандидата добавить новый риск (например, частичный вывод или нестабильный PSP) и адаптировать тест-стратегию.
- Обсудить, как он боролся бы с флейками при росте E2E-набора.
- Проверить, как кандидат приоритизирует баги по влиянию на бизнес.
