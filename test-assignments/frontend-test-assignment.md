# Тестовое задание: Frontend Developer (React + Next.js)

## Формат и цель

Это короткое тестовое на **2-3 часа**.  
Проверяем инженерное мышление в критичном UI-флоу: устойчивость, безопасность и ясность архитектуры.

## Контекст

Нужно реализовать страницу `Withdraw`, которая работает с API вывода средств.

## Core (обязательно, 2-3 часа)

### 1) Страница Withdraw

Поля:
- `amount` (> 0);
- `destination`;
- `confirm` (checkbox).

Требования:
- submit доступен только при валидной форме;
- во время запроса submit disabled.

### 2) API-интеграция

Использовать:
- `POST /v1/withdrawals`
- `GET /v1/withdrawals/{id}`

Поведение:
- отправляется `idempotency_key`;
- `409` показывается понятным текстом;
- при сетевой ошибке есть retry без потери введенных данных;
- после успеха отображается созданная заявка и статус.

### 3) Устойчивость UI

- защита от двойного submit;
- состояния `idle/loading/success/error`.

### 4) Минимум по архитектуре и безопасности

- Next.js App Router + TypeScript;
- Zustand (или аналог) для состояния;
- без небезопасного рендера HTML;
- не хранить access token в `localStorage` (если auth моковая, описать прод-подход в README).

### 5) Что сдать

- исходный код;
- `README.md` (как запустить + ключевые решения);
- тесты.

### 6) Минимальные тесты

- 1 тест happy-path submit;
- 1 тест ошибки API;
- 1 тест защиты от двойного submit.

## Optional (по желанию, +1-2 часа)

- восстановление последней заявки после reload (например, до 5 минут);
- оптимизации производительности (мемоизация/снижение лишних рендеров);
- дополнительный E2E-тест.

## Технические ограничения

- Next.js 14+;
- TypeScript;
- реальная Web3-интеграция не требуется (можно mock).

## Что можно упростить

- один пользователь;
- одна валюта (`USDT`);
- без полноценной auth-системы.
